This multi-vendor marketplace is built with **Next.js 15**, **Payload CMS 3**, **tRPC**, and **shadcn/ui**. The project follows a modular architecture with clear separation of concerns.

## Directory Overview

```
src/
├── app/                    # Next.js App Router
│   ├── (app)/             # Main application routes
│   │   ├── (auth)/        # Authentication pages
│   │   ├── (home)/        # Public marketing pages
│   │   ├── (tenants)/     # Vendor store pages
│   │   └── api/trpc/      # tRPC API endpoints
│   └── (payload)/         # Payload CMS admin
├── collections/            # Payload CMS collections
├── components/            # React components
│   └── ui/               # shadcn/ui components
├── hooks/                # Custom React hooks
├── lib/                  # Utility functions
├── modules/              # Feature modules
│   ├── auth/            # Authentication
│   ├── categories/      # Category management
│   ├── checkout/        # Shopping cart & checkout
│   ├── home/           # Homepage components
│   ├── products/       # Product catalog
│   ├── tags/           # Product tags
│   └── tenants/        # Vendor stores
└── trpc/                # tRPC configuration
    └── routers/         # API routers
```

## App Router Structure

The application uses Next.js App Router with route groups for organization:

### (app) Route Group

Main application routes accessible to users:

- **(auth)/** - Sign in and sign up pages at `/sign-in` and `/sign-up`
- **(home)/** - Public pages like homepage, about, features, pricing, and category browsing
- **(tenants)/tenants/[slug]/** - Individual vendor storefronts with dynamic routing
  - `(home)/` - Store homepage and product listing
  - `(checkout)/` - Shopping cart and checkout flow

### (payload) Route Group

Payload CMS admin interface and APIs:

- **admin/[[...segments]]/** - CMS admin dashboard at `/admin`
- **api/[...slug]/** - Payload REST API
- **api/graphql/** - Payload GraphQL API
- **api/graphql-playground/** - GraphQL playground interface

### tRPC API Routes

Type-safe API endpoints:

```typescript
// src/app/(app)/api/trpc/[trpc]/route.ts
import { fetchRequestHandler } from "@trpc/server/adapters/fetch";
import { createTRPCContext } from "@/trpc/init";
import { appRouter } from "@/trpc/routers/_app";

const handler = (req: Request) =>
  fetchRequestHandler({
    endpoint: "/api/trpc",
    req,
    router: appRouter,
    createContext: createTRPCContext,
  });

export { handler as GET, handler as POST };
```

## Module Architecture

Each feature module follows a consistent structure:

```
modules/[feature]/
├── server/
│   └── procedures.ts    # tRPC procedures
├── ui/
│   ├── components/     # Feature-specific components
│   └── views/          # Page-level views
├── hooks/              # Feature-specific hooks
└── store/              # State management (Zustand)
```

## Path Aliases

The project uses TypeScript path aliases for cleaner imports:

- `@/*` - Maps to `src/*`
- `@payload-config` - Maps to `src/payload.config.ts`

```typescript
// Instead of relative imports
import { Button } from "../../components/ui/button";

// Use path aliases
import { Button } from "@/components/ui/button";
```
